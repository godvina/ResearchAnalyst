# Requirements Document

## Introduction

The Investigative Intelligence Platform currently requires investigators to manually search, query, and explore evidence to discover leads and patterns. There is no automated mechanism that continuously monitors ingested data and graph relationships to proactively surface new intelligence. The Intelligence Trawler is a background analysis engine that periodically scans case data — Neptune graph connections, OpenSearch document indices, entity co-occurrences, and temporal patterns — to detect noteworthy changes and generate alerts. It also optionally trawls external public sources (OSINT) via the existing AIResearchAgent to discover new publicly available intelligence about case entities. When new evidence is ingested, new entity connections form, pattern thresholds are crossed, or external sources surface relevant information, the Trawler creates prioritized alerts that appear in the investigator UI. This extends the existing pattern discovery and cross-case services with a scheduled, event-driven layer that pushes intelligence to investigators rather than waiting for them to pull it.

## Glossary

- **Trawler_Engine**: The backend Python service that runs trawl scans against Neptune graph data, OpenSearch indices, and Aurora metadata to detect new intelligence leads and pattern changes for a case.
- **Trawl_Scan**: A single execution of the Trawler_Engine against one case, producing zero or more Trawler_Alerts.
- **Trawler_Alert**: A structured intelligence notification generated by a Trawl_Scan, containing a type, severity, title, summary, supporting evidence references, and affected entity names.
- **Alert_Severity**: The priority level of a Trawler_Alert: critical, high, medium, or low.
- **Alert_Type**: The category of intelligence detected: new_connection, pattern_change, entity_spike, new_evidence_match, cross_case_overlap, temporal_anomaly, network_expansion, or external_lead.
- **Alert_Panel**: A UI panel in the investigator frontend that displays Trawler_Alerts for the active case, with filtering, dismissal, and drill-down capabilities.
- **Alert_Badge**: A notification count badge displayed on the Case Dashboard tab showing the number of unread Trawler_Alerts.
- **Alert_Store**: The Aurora database table that persists Trawler_Alerts with read/dismissed status per case.
- **Trawl_Config**: Per-case configuration controlling which alert types are enabled, severity thresholds, and scan frequency preferences.
- **Investigator_UI**: The single-page frontend application at `src/frontend/investigator.html`.
- **Pattern_Discovery_Service**: The existing service (`src/services/pattern_discovery_service.py`) that discovers graph, vector, financial, communication, geographic, and temporal patterns.
- **Cross_Case_Service**: The existing service (`src/services/cross_case_service.py`) that detects shared entities and overlaps across cases.
- **Neptune_Graph**: The Amazon Neptune graph database storing entity relationships and connections.
- **OpenSearch_Index**: The Amazon OpenSearch Serverless collection storing document embeddings and text for semantic search.
- **External_Trawl**: An OSINT scan phase within a Trawl_Scan that uses the existing AIResearchAgent and InvestigativeSearchService to check external/public sources for new information relevant to tracked entities and case keywords.
- **OSINT_Alert**: A Trawler_Alert of type external_lead generated when external source scanning discovers publicly available information that correlates with or extends internal case evidence.

## Requirements

### Requirement 1: Trawler Engine Core Scan

**User Story:** As an investigator, I want the system to automatically scan my case data for new intelligence leads, so that I am alerted to important developments without manually searching.

#### Acceptance Criteria

1. WHEN a Trawl_Scan is triggered for a case, THE Trawler_Engine SHALL query the Neptune_Graph for new entity connections added since the last scan timestamp.
2. WHEN a Trawl_Scan is triggered for a case, THE Trawler_Engine SHALL query the OpenSearch_Index for newly ingested documents that match tracked entities or case keywords.
3. WHEN a Trawl_Scan is triggered for a case, THE Trawler_Engine SHALL invoke the Pattern_Discovery_Service to compare current pattern scores against the previous scan's baseline scores.
4. WHEN a Trawl_Scan is triggered for a case, THE Trawler_Engine SHALL invoke the Cross_Case_Service scan_for_overlaps method to detect new cross-case entity matches.
5. WHEN a Trawl_Scan completes, THE Trawler_Engine SHALL store the scan timestamp and pattern baseline in Aurora for use as the comparison point in the next scan.
6. THE Trawler_Engine SHALL complete a Trawl_Scan for a single case within 30 seconds under normal data volumes (fewer than 10,000 graph nodes).

### Requirement 2: Alert Generation Rules

**User Story:** As an investigator, I want alerts generated only when meaningful intelligence is detected, so that I am not overwhelmed by noise.

#### Acceptance Criteria

1. WHEN the Trawler_Engine detects a new entity connection in Neptune_Graph that links two previously unconnected entities with 3 or more shared evidence documents, THE Trawler_Engine SHALL generate a Trawler_Alert of type new_connection with severity high.
2. WHEN the Trawler_Engine detects that a pattern score from Pattern_Discovery_Service has increased by more than 25% compared to the previous baseline, THE Trawler_Engine SHALL generate a Trawler_Alert of type pattern_change.
3. WHEN the Trawler_Engine detects that a single entity appears in 5 or more newly ingested documents since the last scan, THE Trawler_Engine SHALL generate a Trawler_Alert of type entity_spike.
4. WHEN the Trawler_Engine detects a new cross-case entity overlap via Cross_Case_Service, THE Trawler_Engine SHALL generate a Trawler_Alert of type cross_case_overlap with severity high.
5. WHEN the Trawler_Engine detects 3 or more new connections to a single entity since the last scan, THE Trawler_Engine SHALL generate a Trawler_Alert of type network_expansion.
6. WHEN the Trawler_Engine detects a cluster of events within a 48-hour window involving 3 or more tracked entities, THE Trawler_Engine SHALL generate a Trawler_Alert of type temporal_anomaly.
7. THE Trawler_Engine SHALL assign Alert_Severity based on the number of supporting evidence references: critical for 10 or more, high for 5-9, medium for 3-4, low for 1-2.

### Requirement 3: Alert Persistence

**User Story:** As an investigator, I want alerts stored in the database so that I can review them across sessions and share them with team members.

#### Acceptance Criteria

1. THE Alert_Store SHALL persist each Trawler_Alert with fields: alert_id (UUID), case_id, alert_type, severity, title, summary, entity_names (JSON array), evidence_refs (JSON array), source_type (enum: "internal" or "osint"), is_read (boolean), is_dismissed (boolean), created_at (timestamp), and scan_id (UUID).
2. WHEN a Trawler_Alert is generated, THE Trawler_Engine SHALL insert the alert into the Alert_Store with is_read set to false, is_dismissed set to false, and source_type set to "internal" for alerts from Neptune/OpenSearch/pattern scans or "osint" for alerts from external source trawling.
3. THE Alert_Store SHALL support querying alerts by case_id, filtering by alert_type, severity, source_type, is_read, and is_dismissed, and sorting by created_at or severity.
4. IF the Alert_Store insert fails, THEN THE Trawler_Engine SHALL log the error and continue processing remaining alerts from the current scan without stopping.
5. EACH entry in the evidence_refs JSON array SHALL include fields: ref_id (string), ref_type (enum: "document", "graph_edge", "external_url"), source_label (string — e.g. document filename, Neptune edge ID, or external URL/domain), and excerpt (string — relevant text snippet, max 500 chars).

### Requirement 4: Trawl Scan Trigger via API

**User Story:** As an investigator, I want to trigger a trawl scan on demand from the UI, so that I can check for new intelligence whenever I want without waiting for a scheduled run.

#### Acceptance Criteria

1. WHEN the investigator sends a POST request to `/v1/cases/{case_id}/trawl`, THE API SHALL invoke the Trawler_Engine to run a Trawl_Scan for the specified case and return the scan results.
2. WHEN the investigator sends a GET request to `/v1/cases/{case_id}/alerts`, THE API SHALL return the list of Trawler_Alerts for the case, supporting query parameters for alert_type, severity, is_read, and limit.
3. WHEN the investigator sends a PATCH request to `/v1/cases/{case_id}/alerts/{alert_id}`, THE API SHALL update the alert's is_read or is_dismissed status.
4. WHEN the investigator sends a POST request to `/v1/cases/{case_id}/alerts/{alert_id}/investigate`, THE API SHALL mark the alert as read and return the alert's entity_names and evidence_refs formatted for use by the existing drill-down and search APIs.
5. IF the specified case_id does not exist, THEN THE API SHALL return HTTP 404 with a descriptive error message.

### Requirement 5: Alert Panel in Investigator UI

**User Story:** As an investigator, I want to see alerts in a dedicated panel within the Case Dashboard, so that I can quickly review and act on new intelligence.

#### Acceptance Criteria

1. WHEN the Case Dashboard tab loads and the active case has unread alerts, THE Alert_Panel SHALL render as a collapsible section at the top of the dashboard showing the most recent alerts.
2. THE Alert_Panel SHALL display each Trawler_Alert as a card showing: severity icon (🔴 critical, 🟠 high, 🟡 medium, 🔵 low), source badge ("🏠 Internal" or "🌐 External"), alert type badge, title, summary (truncated to 2 lines), entity count, and timestamp.
3. WHEN the investigator clicks on a Trawler_Alert card, THE Alert_Panel SHALL expand the card to show the full summary, list of entity_names as clickable entity links, and list of evidence_refs as clickable links with each ref displaying its ref_type icon (📄 document, 🕸️ graph_edge, 🌐 external_url) and source_label.
4. THE Alert_Panel SHALL provide filter buttons for alert severity levels, a source filter toggle (All / Internal / External), and a toggle to show/hide dismissed alerts.
5. WHEN the investigator clicks "Dismiss" on a Trawler_Alert card, THE Alert_Panel SHALL mark the alert as dismissed via the PATCH API and visually fade the card.
6. WHEN the investigator clicks "Investigate" on a Trawler_Alert card, THE Investigator_UI SHALL open the first entity from the alert in the Entity Dossier panel and mark the alert as read.
7. THE Alert_Panel SHALL use the existing dark theme styling consistent with the Investigator_UI (background: #1a2332, borders: #2d3748, severity colors for accent).

### Requirement 6: Alert Badge Notification

**User Story:** As an investigator, I want to see a badge count of unread alerts on the Case Dashboard tab, so that I know when new intelligence is available without opening the dashboard.

#### Acceptance Criteria

1. WHEN the active case has unread (is_read = false, is_dismissed = false) Trawler_Alerts, THE Investigator_UI SHALL display an Alert_Badge on the Case Dashboard tab showing the count of unread alerts.
2. WHEN the unread alert count changes (alert read, dismissed, or new alerts generated), THE Alert_Badge SHALL update the count without requiring a full page reload.
3. WHEN the unread alert count is zero, THE Alert_Badge SHALL be hidden.
4. THE Alert_Badge SHALL use a red background (#e53e3e) with white text, positioned at the top-right corner of the Case Dashboard tab label.

### Requirement 7: Trawl Configuration Per Case

**User Story:** As an investigator, I want to configure which types of alerts I receive for each case, so that I can focus the trawler on the intelligence most relevant to my investigation.

#### Acceptance Criteria

1. THE Alert_Panel SHALL include a "⚙️ Trawl Settings" button that opens a configuration modal.
2. THE Trawl_Config modal SHALL display toggles for each Alert_Type (new_connection, pattern_change, entity_spike, new_evidence_match, cross_case_overlap, temporal_anomaly, network_expansion), all enabled by default.
3. THE Trawl_Config modal SHALL display a minimum severity threshold selector (critical, high, medium, low) that filters which alerts are shown, defaulting to low (show all).
4. WHEN the investigator saves Trawl_Config changes, THE Investigator_UI SHALL persist the configuration to localStorage using the key `trawlConfig_{caseId}` and send the configuration to the API via PUT `/v1/cases/{case_id}/trawl-config`.
5. WHEN the Trawler_Engine runs a scan, THE Trawler_Engine SHALL respect the Trawl_Config for the case, skipping disabled alert types and suppressing alerts below the minimum severity threshold.

### Requirement 8: Alert-to-Investigation Workflow

**User Story:** As an investigator, I want to seamlessly transition from an alert to a full investigation action, so that I can act on intelligence leads without manual data re-entry.

#### Acceptance Criteria

1. WHEN the investigator clicks "Investigate" on a new_connection alert, THE Investigator_UI SHALL navigate to the Dashboard tab and open the Entity Dossier for the primary entity in the connection.
2. WHEN the investigator clicks "Investigate" on a cross_case_overlap alert, THE Investigator_UI SHALL navigate to the Dashboard tab and trigger a cross-case search for the overlapping entity.
3. WHEN the investigator clicks "Investigate" on an entity_spike alert, THE Investigator_UI SHALL navigate to the Evidence Library tab with the entity name pre-filled in the search filter.
4. WHEN the investigator clicks "Investigate" on a temporal_anomaly alert, THE Investigator_UI SHALL navigate to the Timeline tab with the relevant date range highlighted.
5. WHEN the investigator clicks "Investigate" on a network_expansion alert, THE Investigator_UI SHALL navigate to the Dashboard tab and open the Entity Dossier for the expanding entity.
6. WHEN the investigator clicks "Save to Notebook" on any expanded alert card, THE Investigator_UI SHALL save the alert as a finding via the existing FindingsService with finding_type set to "trawler_alert".

### Requirement 9: Scan History and Audit Trail

**User Story:** As an investigator, I want to see a history of trawl scans and their results, so that I can track what the system has analyzed and when.

#### Acceptance Criteria

1. THE Trawler_Engine SHALL persist each Trawl_Scan execution as a record in Aurora with fields: scan_id (UUID), case_id, started_at (timestamp), completed_at (timestamp), alerts_generated (integer), scan_status (completed, failed, partial).
2. WHEN the investigator sends a GET request to `/v1/cases/{case_id}/trawl/history`, THE API SHALL return the list of Trawl_Scan records for the case, sorted by started_at descending, limited to the most recent 50 scans.
3. THE Alert_Panel SHALL include a "📊 Scan History" link that displays the recent scan history in a collapsible section showing scan timestamp, duration, alert count, and status.
4. IF a Trawl_Scan fails partway through, THEN THE Trawler_Engine SHALL record the scan with status "partial", persist any alerts generated before the failure, and log the error.

### Requirement 10: New Evidence Ingestion Trigger

**User Story:** As an investigator, I want the trawler to automatically scan when new evidence is ingested into my case, so that I get immediate alerts about intelligence from new data.

#### Acceptance Criteria

1. WHEN new documents are ingested into a case via the ingestion pipeline, THE Trawler_Engine SHALL receive a trigger to run a targeted Trawl_Scan focused on the newly ingested documents.
2. WHEN a targeted Trawl_Scan runs, THE Trawler_Engine SHALL compare entities extracted from the new documents against existing graph entities to detect new connections.
3. WHEN a targeted Trawl_Scan runs, THE Trawler_Engine SHALL check if any newly extracted entities match tracked entities for the case.
4. THE targeted Trawl_Scan SHALL complete within 15 seconds for a batch of up to 50 newly ingested documents.
5. IF the ingestion trigger fails to invoke the Trawler_Engine, THEN the next scheduled or manual Trawl_Scan SHALL detect the same intelligence from the ingested data.

### Requirement 11: Alert Deduplication

**User Story:** As an investigator, I want the trawler to avoid generating duplicate alerts for the same intelligence, so that the alert list remains clean and actionable.

#### Acceptance Criteria

1. WHEN the Trawler_Engine generates a Trawler_Alert, THE Trawler_Engine SHALL check the Alert_Store for existing non-dismissed alerts with the same alert_type, case_id, and overlapping entity_names within the past 7 days.
2. IF a matching existing alert is found, THEN THE Trawler_Engine SHALL update the existing alert's evidence_refs and summary with the new information instead of creating a duplicate alert.
3. WHEN an existing alert is updated with new evidence, THE Trawler_Engine SHALL reset the alert's is_read status to false and update the created_at timestamp so the alert resurfaces.
4. THE Trawler_Engine SHALL log deduplicated alerts at debug level for audit purposes.

### Requirement 12: Non-Interference with Existing UI

**User Story:** As an investigator, I want the alert panel and badge to integrate seamlessly with the existing UI without breaking any current functionality.

#### Acceptance Criteria

1. THE Alert_Panel SHALL be implemented as self-contained HTML, CSS, and JavaScript appended to the existing investigator.html file.
2. THE Alert_Panel SHALL not modify any existing HTML elements, CSS classes, or JavaScript functions in the Investigator_UI.
3. THE Alert_Panel SHALL use a z-index below the Entity Dossier panel (z-index 200) so that the Entity Dossier renders above the Alert_Panel when both are visible.
4. THE Alert_Badge SHALL be appended to the existing Case Dashboard tab element without altering the tab's existing onclick handler or styling.

### Requirement 13: External Source Trawling (OSINT)

**User Story:** As an investigator, I want the trawler to scan external public sources for new information about my case entities, so that I am alerted when publicly available intelligence emerges that could strengthen or redirect my investigation.

#### Acceptance Criteria

1. WHEN a Trawl_Scan runs with external trawling enabled in Trawl_Config, THE Trawler_Engine SHALL invoke the existing AIResearchAgent to perform OSINT research on the top 5 tracked entities for the case.
2. WHEN the AIResearchAgent returns external research results, THE Trawler_Engine SHALL invoke the InvestigativeSearchService cross-reference report generator to compare external findings against internal case evidence.
3. WHEN the cross-reference report identifies findings categorized as "external_only" (information found externally but not present in internal evidence), THE Trawler_Engine SHALL generate a Trawler_Alert of type external_lead with the external finding as the summary and the source URL as an evidence reference.
4. WHEN the cross-reference report identifies findings categorized as "confirmed_internally" (external information that corroborates internal evidence), THE Trawler_Engine SHALL generate a Trawler_Alert of type external_lead with severity medium and a summary noting the corroboration.
5. THE Trawl_Config modal SHALL include an "External Sources" toggle (disabled by default) that enables or disables the External_Trawl phase of scans.
6. WHEN external trawling is enabled, THE Trawler_Engine SHALL limit external research to a maximum of 5 entities per scan and a total execution time of 60 seconds for the external phase to avoid excessive Bedrock costs.
7. THE Trawler_Engine SHALL populate evidence_refs for external_lead alerts with ref_type set to "external_url" and source_label set to the actual source URL or domain name returned by the AIResearchAgent, so that investigators can trace each finding back to its public source.
8. WHEN the investigator clicks "Investigate" on an external_lead alert, THE Investigator_UI SHALL navigate to the Dashboard tab and trigger an internal+external investigative search for the entity, pre-populating the search with the external finding context.
