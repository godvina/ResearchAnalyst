# Requirements Document — Lead to Investigation

## Introduction

This feature enables the Research Analyst platform to accept structured lead JSON from an external lead-finding application and automatically create a pre-populated investigation case. The feature has two phases:

- **Phase A (Lead Ingestion + AI Research Agent):** Accept lead JSON, create a Matter/Collection in Aurora, seed Neptune with subjects and connections, run an AI Research Agent (Bedrock Haiku) to gather public information about each subject, feed research through the existing Step Functions ingestion pipeline, and present the investigator with a pre-populated conspiracy graph.

- **Phase C (Autonomous Investigation Agent):** A long-running AI agent that reasons about the case, identifies gaps in the Neptune graph, generates and scores hypotheses, plans and executes research tasks in a loop, and suggests next investigative steps to the human analyst. Phase C is aspirational — the spec defines requirements but the user will implement it independently.

Both phases extend existing infrastructure (Aurora, Neptune, S3, Lambda mega-dispatcher, Step Functions pipeline, Bedrock Haiku). Per lessons-learned: all changes EXTEND existing code, never REPLACE working modules.

## Glossary

- **Lead_Ingestion_Service**: The new service module that validates incoming lead JSON, creates Aurora records (Matter + Collection), and seeds Neptune with subjects and connections from the lead.
- **AI_Research_Agent**: The new service module that uses Bedrock Claude Haiku to generate OSINT research queries for each subject in a lead, fetch public information (SEC filings, court records, news articles), and produce synthetic research documents for pipeline ingestion.
- **Autonomous_Investigation_Agent**: The Phase C long-running agent (ECS task or Step Functions workflow) that analyzes Neptune graph state, generates hypotheses, plans research tasks, and iterates until a confidence threshold or budget is exhausted.
- **Lead_JSON**: The structured JSON payload submitted by the external lead-finding application, containing subjects, connections, evidence hints, OSINT directives, tags, and statutes.
- **Subject**: A person or organization identified in the Lead_JSON as a target of investigation.
- **Connection**: A relationship between two Subjects in the Lead_JSON, with a type, confidence score, and source attribution.
- **Evidence_Hint**: A reference to a public document (filing, article, court record) included in the Lead_JSON that the AI_Research_Agent should retrieve and process.
- **OSINT_Directive**: A natural-language instruction in the Lead_JSON telling the AI_Research_Agent what public information to search for.
- **Research_Document**: A synthetic document produced by the AI_Research_Agent containing gathered public information about a Subject, formatted for ingestion through the existing Step Functions pipeline.
- **Hypothesis**: A scored proposition generated by the Autonomous_Investigation_Agent about a connection or pattern in the Neptune graph.
- **Investigation_Loop**: The iterative cycle in Phase C where the Autonomous_Investigation_Agent analyzes the graph, generates hypotheses, executes research, ingests results, and re-analyzes.
- **Mega_Dispatcher**: The existing `case_files.py` Lambda handler that routes all API requests via `{proxy+}` catch-all.
- **Matter**: An investigation case in Aurora (from migration 006) scoped to an organization.
- **Collection**: A tracked data load within a Matter, following the staging → processing → qa_review → promoted lifecycle.
- **Neptune_Graph**: The existing Neptune Serverless knowledge graph storing entity nodes and relationship edges, queried via Gremlin HTTP API.
- **Step_Functions_Pipeline**: The existing ingestion pipeline that processes documents through parse → extract → embed → store artifact → graph load → update status.

## Requirements

### Requirement 1: Lead JSON Validation

**User Story:** As an external lead-finding application, I want to submit structured lead JSON to the Research Analyst platform, so that an investigation case is automatically created from the lead data.

#### Acceptance Criteria

1. WHEN a Lead_JSON payload is submitted to `POST /leads/ingest`, THE Lead_Ingestion_Service SHALL validate the payload against the lead JSON schema and return a 400 error with descriptive field-level messages for any schema violation.
2. THE Lead_Ingestion_Service SHALL require the following fields in every Lead_JSON payload: `lead_id`, `classification`, `title`, `summary`, `subjects` (with at least one entry), and `connections`.
3. WHEN a Lead_JSON payload contains a `lead_id` that has already been ingested, THE Lead_Ingestion_Service SHALL return a 409 Conflict response with the existing Matter ID.
4. THE Lead_Ingestion_Service SHALL validate that every Subject in the `subjects` array has a non-empty `name` and a `type` value of either `person` or `organization`.
5. THE Lead_Ingestion_Service SHALL validate that every Connection in the `connections` array references Subject names that exist in the `subjects` array via the `from` and `to` fields.
6. WHEN a valid Lead_JSON payload is submitted, THE Lead_Ingestion_Service SHALL return a 202 Accepted response containing the created `matter_id`, `collection_id`, and a `status` of `processing`.

### Requirement 2: Matter and Collection Creation from Lead

**User Story:** As an investigator, I want a lead to automatically create a Matter and Collection in Aurora, so that I can find and manage the investigation through the existing Matter hierarchy.

#### Acceptance Criteria

1. WHEN a valid Lead_JSON is accepted, THE Lead_Ingestion_Service SHALL create a new Matter in Aurora with `matter_name` derived from the Lead_JSON `title`, `description` from the Lead_JSON `summary`, `matter_type` set to `lead_investigation`, and `status` set to `processing`.
2. WHEN a Matter is created from a lead, THE Lead_Ingestion_Service SHALL create a Collection within that Matter with `collection_name` set to `Lead: {lead_id}` and `source_description` containing the `source_app`, `classification`, and `subcategory` from the Lead_JSON.
3. WHEN a Matter is created from a lead, THE Lead_Ingestion_Service SHALL store the complete Lead_JSON as a JSON file in S3 at the Collection's S3 prefix under `lead_data/lead.json`.
4. THE Lead_Ingestion_Service SHALL store the `lead_id`, `classification`, `subcategory`, `priority`, `tags`, and `statutes` as JSONB metadata on the Matter row in a new `lead_metadata` column.

### Requirement 3: Neptune Graph Seeding from Lead Subjects

**User Story:** As an investigator, I want the subjects and connections from a lead to appear immediately in the Neptune graph, so that I see the known network structure before AI research begins.

#### Acceptance Criteria

1. WHEN a Matter is created from a lead, THE Lead_Ingestion_Service SHALL create a Neptune entity node for each Subject in the Lead_JSON, using the Matter's `neptune_subgraph_label` as the node label, with properties: `canonical_name` from Subject `name`, `entity_type` from Subject `type`, `confidence` set to 1.0, `occurrence_count` set to 1, and `case_file_id` set to the Matter ID.
2. WHEN a Matter is created from a lead, THE Lead_Ingestion_Service SHALL create a Neptune `RELATED_TO` edge for each Connection in the Lead_JSON, with properties: `relationship_type` from Connection `relationship`, `confidence` from Connection `confidence`, and `source_document_ref` set to `lead:{lead_id}`.
3. WHEN a Subject has `aliases` in the Lead_JSON, THE Lead_Ingestion_Service SHALL store aliases as a multi-value property `aliases` on the Neptune entity node.
4. WHEN a Subject has `identifiers` in the Lead_JSON, THE Lead_Ingestion_Service SHALL store each identifier key-value pair as a property on the Neptune entity node with prefix `id_` (e.g., `id_ein`, `id_ssn_last4`).
5. IF the Neptune graph seeding fails for any subject or connection, THEN THE Lead_Ingestion_Service SHALL log the failure, continue processing remaining subjects and connections, and include the failure count in the API response.

### Requirement 4: AI Research Agent — Subject Research

**User Story:** As an investigator, I want an AI agent to automatically gather public information about each subject in a lead, so that the conspiracy graph is enriched with real-world data before I begin my analysis.

#### Acceptance Criteria

1. WHEN Neptune graph seeding completes, THE AI_Research_Agent SHALL generate research queries for each Subject using Bedrock Claude Haiku (model ID: `anthropic.claude-3-haiku-20240307-v1:0`), incorporating the Subject's name, type, role, aliases, identifiers, and any OSINT directives from the Lead_JSON.
2. THE AI_Research_Agent SHALL produce one Research_Document per Subject containing the gathered public information, structured with sections for each information source (SEC filings, court records, news articles, OFAC/SDN matches).
3. WHEN Evidence_Hints are present in the Lead_JSON, THE AI_Research_Agent SHALL include the evidence hint descriptions and URLs in the Research_Document for the relevant Subject.
4. THE AI_Research_Agent SHALL store each Research_Document as a text file in S3 at the Collection's S3 prefix under `research/{subject_name_slug}.txt`.
5. WHEN all Research_Documents are generated, THE AI_Research_Agent SHALL trigger the existing Step_Functions_Pipeline for the Collection, passing the list of Research_Document IDs for processing through entity extraction, embedding generation, and graph loading.
6. IF Bedrock returns an error or timeout for a Subject research query, THEN THE AI_Research_Agent SHALL retry up to 2 times with exponential backoff, and if all retries fail, log the failure and continue with remaining Subjects.

### Requirement 5: Pipeline Integration for Research Documents

**User Story:** As an investigator, I want AI-gathered research to flow through the same ingestion pipeline as manually uploaded documents, so that entities and relationships are extracted consistently and appear in the Neptune graph.

#### Acceptance Criteria

1. THE AI_Research_Agent SHALL register each Research_Document in the Aurora `documents` table with `case_file_id` set to the Matter ID, `collection_id` set to the Collection ID, `source_filename` set to the research file name, and `source_metadata` containing `{"source": "ai_research_agent", "subject": "<subject_name>", "lead_id": "<lead_id>"}`.
2. WHEN the Step_Functions_Pipeline processes a Research_Document, THE Step_Functions_Pipeline SHALL extract entities and relationships using the existing entity extraction service with chunking and merge logic.
3. WHEN the Step_Functions_Pipeline completes for all Research_Documents, THE Lead_Ingestion_Service SHALL update the Matter status to `indexed` and the Collection status to `promoted`.
4. WHEN the Step_Functions_Pipeline fails for a Research_Document, THE Step_Functions_Pipeline SHALL continue processing remaining documents and record the failure in the Collection's error log.

### Requirement 6: Lead Ingestion API Route

**User Story:** As a developer integrating the lead-finding application, I want a documented REST API endpoint for submitting leads, so that I can programmatically create investigations.

#### Acceptance Criteria

1. THE Mega_Dispatcher SHALL route `POST /leads/ingest` requests to the Lead_Ingestion_Service handler.
2. THE Mega_Dispatcher SHALL route `GET /leads/{lead_id}/status` requests to a handler that returns the current processing status of a lead, including `matter_id`, `collection_id`, `status`, `subjects_seeded`, `research_documents_generated`, and `pipeline_status`.
3. THE Mega_Dispatcher SHALL route `GET /matters/{id}/lead` requests to a handler that returns the original Lead_JSON metadata stored on the Matter.
4. THE Lead_Ingestion_Service SHALL use the default organization ID for all Matter and Collection creation (consistent with existing `ACCESS_CONTROL_ENABLED=false` behavior).

### Requirement 7: Lead Processing Status Tracking

**User Story:** As an external application, I want to poll the status of a submitted lead, so that I know when the investigation case is ready for an analyst.

#### Acceptance Criteria

1. THE Lead_Ingestion_Service SHALL track lead processing through the following statuses: `accepted`, `seeding_graph`, `researching`, `pipeline_running`, `indexed`, `error`.
2. WHEN the lead processing status changes, THE Lead_Ingestion_Service SHALL update a `lead_status` column on the Matter row in Aurora and update `last_activity` timestamp.
3. WHEN the lead processing encounters an error at any phase, THE Lead_Ingestion_Service SHALL set the lead status to `error` and store a descriptive error message in the Matter's `error_details` column.
4. THE `GET /leads/{lead_id}/status` endpoint SHALL return the response within 200ms for any lead that has been ingested.


### Requirement 8: Database Schema Extensions for Lead Tracking

**User Story:** As a platform operator, I want lead metadata persisted in Aurora, so that leads are traceable from ingestion through investigation completion.

#### Acceptance Criteria

1. THE Lead_Ingestion_Service SHALL add a `lead_metadata` JSONB column to the `matters` table via a new migration file (`008_lead_to_investigation.sql`).
2. THE Lead_Ingestion_Service SHALL add a `lead_status` TEXT column to the `matters` table with a default value of `null` (non-lead matters have no lead status).
3. THE Lead_Ingestion_Service SHALL add a `lead_id` TEXT column to the `matters` table with a unique index, allowing lookup of matters by lead ID.
4. THE migration SHALL be additive only — no existing columns or tables are modified or dropped.

### Requirement 9: Autonomous Investigation Agent — Graph Analysis (Phase C — Aspirational)

**User Story:** As an investigator, I want an AI agent to autonomously analyze the Neptune graph, identify gaps, and suggest next steps, so that routine investigative grunt work is automated.

#### Acceptance Criteria

1. WHEN an investigator activates the Autonomous_Investigation_Agent for a Matter, THE Autonomous_Investigation_Agent SHALL query the Neptune_Graph to build a current state snapshot including all entity nodes, relationship edges, centrality scores, and community clusters for the Matter's subgraph label.
2. THE Autonomous_Investigation_Agent SHALL use Bedrock Claude Haiku to analyze the graph snapshot and identify gaps: entities with few connections, missing expected relationship types, and subjects lacking corroborating evidence.
3. THE Autonomous_Investigation_Agent SHALL generate scored Hypotheses about connections between subjects, with each Hypothesis containing: a natural-language statement, a confidence score (0.0–1.0), supporting evidence references, and a list of research tasks needed to confirm or deny the Hypothesis.
4. WHILE the Investigation_Loop is active, THE Autonomous_Investigation_Agent SHALL plan research tasks to confirm or deny each Hypothesis, execute the research via the AI_Research_Agent, ingest results through the Step_Functions_Pipeline, and re-analyze the updated graph.
5. WHEN a research task produces results that confirm or deny a Hypothesis, THE Autonomous_Investigation_Agent SHALL update the Hypothesis score to one of: `confirmed` (confidence >= 0.8), `denied` (confidence <= 0.2), or `needs_more_evidence` (0.2 < confidence < 0.8).

### Requirement 10: Autonomous Investigation Agent — Loop Control (Phase C — Aspirational)

**User Story:** As an investigator, I want the autonomous agent to stop when it reaches a confidence threshold or exhausts its budget, so that compute costs are bounded and I maintain oversight.

#### Acceptance Criteria

1. THE Autonomous_Investigation_Agent SHALL accept configuration parameters: `max_iterations` (default: 10), `confidence_threshold` (default: 0.8), `max_research_documents` (default: 50), and `timeout_minutes` (default: 120).
2. WHEN the Investigation_Loop reaches `max_iterations`, THE Autonomous_Investigation_Agent SHALL stop the loop and produce a final summary report.
3. WHEN all Hypotheses reach `confirmed` or `denied` status, THE Autonomous_Investigation_Agent SHALL stop the loop and produce a final summary report.
4. WHEN the Investigation_Loop exceeds `timeout_minutes`, THE Autonomous_Investigation_Agent SHALL stop the loop, save current state, and produce a partial summary report.
5. WHEN the total Research_Documents generated exceeds `max_research_documents`, THE Autonomous_Investigation_Agent SHALL stop generating new research and produce a summary with remaining unresolved Hypotheses.

### Requirement 11: Autonomous Investigation Agent — Human-in-the-Loop (Phase C — Aspirational)

**User Story:** As an investigator, I want to review the agent's findings and approve or redirect its research direction, so that I maintain control over the investigation strategy.

#### Acceptance Criteria

1. WHEN the Autonomous_Investigation_Agent completes an Investigation_Loop iteration, THE Autonomous_Investigation_Agent SHALL produce an iteration summary containing: new Hypotheses generated, Hypotheses confirmed or denied, research tasks executed, and suggested next steps.
2. THE Autonomous_Investigation_Agent SHALL expose a `GET /matters/{id}/agent/status` endpoint returning the current iteration number, active Hypotheses, and loop state (running, paused, completed, error).
3. THE Autonomous_Investigation_Agent SHALL expose a `POST /matters/{id}/agent/directive` endpoint allowing the investigator to add new OSINT directives, pause the loop, resume the loop, or terminate the loop.
4. WHEN the investigator adds a new directive, THE Autonomous_Investigation_Agent SHALL incorporate the directive into the next iteration's research planning.
5. WHEN the investigator pauses the loop, THE Autonomous_Investigation_Agent SHALL complete the current iteration, save state, and wait for a resume or terminate command.

### Requirement 12: Lead JSON Serialization Round-Trip

**User Story:** As a developer, I want to ensure that Lead_JSON can be stored and retrieved without data loss, so that the original lead data is always available for audit and re-processing.

#### Acceptance Criteria

1. THE Lead_Ingestion_Service SHALL serialize the Lead_JSON to S3 and deserialize it back, producing an object equivalent to the original Lead_JSON (round-trip property).
2. THE Lead_Ingestion_Service SHALL serialize the Lead_JSON `lead_metadata` subset to the Aurora JSONB column and deserialize it back, producing an object equivalent to the original subset (round-trip property).
3. FOR ALL valid Lead_JSON payloads, storing to S3 then reading from S3 then parsing SHALL produce a Lead_JSON object with identical `subjects`, `connections`, `evidence_hints`, `osint_directives`, `tags`, and `statutes` arrays.
