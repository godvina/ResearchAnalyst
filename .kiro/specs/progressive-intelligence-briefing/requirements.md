# Requirements Document

## Introduction

The Progressive Intelligence Briefing replaces the current flat list of 50+ intelligence alert cards in the Alert Panel with a 3-layer progressive disclosure system. Layer 1 is a compact headline bar showing case viability score, delta, and alert count. Layer 2 is an expandable intelligence brief with AI-generated summary and indicator comparison bars. Layer 3 groups alerts by entity cluster instead of showing them individually. The system is designed for investigative analysts who need to quickly answer "did my case get stronger or weaker after new evidence was loaded?" without reading every card.

All three layers work without Bedrock (deterministic fallbacks). The existing `_renderCard` method is reused for individual alert cards within Layer 3 groups. The existing `AlertPanelManager` is extended, not replaced.

Layer 3 includes three enhancements: (1) Investigative Entity Titles that show descriptive, color-coded headers based on dominant alert type, (2) a 4-section BLUF/Key Finding/Critical Gap/Next Action intelligence brief per entity group, and (3) an inline force-directed network graph that fetches entity neighborhoods from Neptune. All three enhancements are deterministic and work without Bedrock.

## Glossary

- **Alert_Panel**: The `AlertPanelManager` JavaScript object in `investigator.html` that renders intelligence alerts in the Case Dashboard tab
- **Headline_Bar**: Layer 1 compact bar showing viability score, delta, alert count, and last scan timestamp
- **Intelligence_Brief**: Layer 2 expandable section containing an AI-generated summary and indicator comparison bars
- **Grouped_Detail_View**: Layer 3 view that organizes alerts into collapsible entity cluster groups
- **Viability_Score**: A 0-100 composite score computed by the Command Center Engine from 5 indicator scores
- **Verdict**: A classification of the viability score: "PURSUE" (67-100), "INVESTIGATE FURTHER" (34-66), or "CLOSE" (0-33)
- **Delta**: The difference in viability score between the two most recent trawl scans
- **Indicator_Bar**: A mini horizontal bar showing before/after values for one of the 5 Command Center indicators
- **Entity_Cluster**: A group of alerts sharing the same first entity name in their `entity_names` JSONB array
- **Investigative_Title**: A descriptive, color-coded entity group header generated by `_buildInvestigativeTitle()` based on the dominant alert type within the cluster
- **Entity_Intel_Brief**: A 4-section structured brief (BLUF/Key Finding/Critical Gap/Next Action) rendered per entity group by `_renderEntityIntelBrief()`
- **Inline_Network_Graph**: An SVG force-directed graph showing an entity's 1-hop neighborhood from Neptune, rendered by `_renderForceGraph()`
- **Briefing_API**: The `GET /case-files/{id}/trawl/briefing` endpoint that returns the AI-generated intelligence brief
- **Impact_API**: The existing `GET /case-files/{id}/trawl/impact` endpoint that returns indicator snapshots and deltas
- **Alerts_API**: The existing `GET /case-files/{id}/alerts` endpoint that returns individual alert objects
- **Fallback_Brief**: A deterministic text summary generated without Bedrock, using alert counts, top entities, and indicator deltas

## Requirements

### Requirement 1: Layer 1 Headline Bar Display

**User Story:** As an investigative analyst, I want to see a single compact headline bar at the top of the alerts panel showing case viability, delta, and alert count, so that I can understand case status at a glance without expanding anything.

#### Acceptance Criteria

1. WHEN the Alert_Panel loads for a case, THE Headline_Bar SHALL display the current Viability_Score as "{score}/100" followed by the Verdict text
2. WHEN the Impact_API returns a non-zero viability delta, THE Headline_Bar SHALL display the Delta as "↑{points} pts — Case STRONGER" for positive deltas or "↓{points} pts — Case WEAKER" for negative deltas
3. WHEN the Impact_API returns a zero viability delta or no previous scan exists, THE Headline_Bar SHALL omit the Delta display
4. THE Headline_Bar SHALL display the total count of non-dismissed alerts as "{count} new findings"
5. THE Headline_Bar SHALL display the timestamp of the most recent completed trawl scan
6. THE Headline_Bar SHALL render using only data from the Impact_API and the cached alert count, requiring zero Bedrock calls
7. THE Headline_Bar SHALL apply color coding to the Viability_Score: green (#48bb78) for PURSUE, amber (#ecc94b) for INVESTIGATE FURTHER, red (#fc8181) for CLOSE
8. THE Headline_Bar SHALL remain visible at all times when the Alert_Panel is rendered, regardless of which layer is expanded

### Requirement 2: Layer 1 Headline Interaction

**User Story:** As an investigative analyst, I want to click the headline bar to expand the intelligence brief, so that I can get a deeper summary without scrolling through individual alerts.

#### Acceptance Criteria

1. WHEN the analyst clicks the Headline_Bar, THE Alert_Panel SHALL expand to show the Intelligence_Brief (Layer 2)
2. WHEN the analyst clicks the Headline_Bar while Layer 2 is visible, THE Alert_Panel SHALL collapse Layer 2 back to the Headline_Bar only
3. THE Headline_Bar SHALL display a visual expand/collapse indicator (▶ when collapsed, ▼ when expanded)

### Requirement 3: Layer 2 Intelligence Brief Content

**User Story:** As an investigative analyst, I want to see an AI-generated intelligence brief summarizing what changed, which entities matter most, and what to do next, so that I can make informed decisions without reading every alert.

#### Acceptance Criteria

1. WHEN Layer 2 expands, THE Intelligence_Brief SHALL display a 3-4 sentence AI-generated summary covering: what changed since the last scan, which entities are most significant, and the recommended next action
2. WHEN the Briefing_API returns successfully, THE Intelligence_Brief SHALL use the AI-generated text from the response
3. WHEN the Briefing_API fails or times out, THE Intelligence_Brief SHALL display a Fallback_Brief generated deterministically from alert counts, top entity names, and indicator deltas
4. THE Intelligence_Brief SHALL display 5 Indicator_Bars showing before/after values for Signal Strength, Corroboration, Network Density, Temporal Coherence, and Prosecution Readiness
5. WHEN the Impact_API provides before and after indicator snapshots, each Indicator_Bar SHALL render a horizontal bar with the before value in muted color and the after value in the indicator's status color
6. THE Intelligence_Brief SHALL display a "View All Findings" button to transition to Layer 3

### Requirement 4: Layer 2 AI Brief Generation (Backend)

**User Story:** As an investigative analyst, I want the intelligence brief to be generated quickly using lightweight AI, so that I do not wait more than a few seconds for the summary.

#### Acceptance Criteria

1. WHEN the Briefing_API receives a request, THE Briefing_API SHALL invoke Bedrock Claude Haiku with a prompt containing the alert summary statistics, top 5 entity names by frequency, and indicator deltas
2. THE Briefing_API SHALL enforce a 3-second timeout on the Bedrock call
3. IF the Bedrock call exceeds the 3-second timeout, THEN THE Briefing_API SHALL return the Fallback_Brief instead
4. IF the Bedrock client is unavailable or returns an error, THEN THE Briefing_API SHALL return the Fallback_Brief instead
5. THE Fallback_Brief SHALL be constructed deterministically as: "{alert_count} new findings detected. Top entities: {top_3_entity_names}. Viability moved from {before_score} to {after_score} ({delta_direction}). {strongest_indicator_name} is the strongest signal at {score}/100."

### Requirement 5: Layer 3 Grouped Detail View

**User Story:** As an investigative analyst, I want alerts grouped by entity cluster instead of a flat list, so that I can focus on one entity network at a time and understand the scope of each cluster.

#### Acceptance Criteria

1. WHEN the analyst clicks "View All Findings" in Layer 2, THE Grouped_Detail_View SHALL replace the flat alert list with alerts organized into Entity_Cluster groups
2. THE Grouped_Detail_View SHALL group alerts by the first element of each alert's `entity_names` array
3. WHEN an alert has an empty `entity_names` array, THE Grouped_Detail_View SHALL place the alert in an "Ungrouped" cluster
4. THE Grouped_Detail_View SHALL display each Entity_Cluster as a collapsible section header showing the entity name and alert count (e.g., "Visoski Network (12 overlaps)")
5. THE Grouped_Detail_View SHALL sort Entity_Cluster groups by alert count descending
6. WHEN the analyst clicks an Entity_Cluster header, THE Grouped_Detail_View SHALL expand the group to show individual alert cards rendered by the existing `_renderCard` method
7. WHEN the analyst clicks an expanded Entity_Cluster header, THE Grouped_Detail_View SHALL collapse the group
8. THE Grouped_Detail_View SHALL perform all grouping logic client-side using the alert data already fetched from the Alerts_API

### Requirement 6: Layer Navigation and State

**User Story:** As an investigative analyst, I want smooth transitions between the 3 layers with clear navigation, so that I can drill down and back up without losing context.

#### Acceptance Criteria

1. WHEN the Alert_Panel first loads, THE Alert_Panel SHALL display Layer 1 (Headline_Bar) in collapsed state with Layer 2 and Layer 3 hidden
2. THE Alert_Panel SHALL maintain the current layer state (1, 2, or 3) across alert data refreshes within the same case session
3. WHEN the analyst is viewing Layer 3, THE Alert_Panel SHALL display a "← Back to Brief" button that returns to Layer 2
4. WHEN the analyst is viewing Layer 2, THE Alert_Panel SHALL allow collapsing back to Layer 1 by clicking the Headline_Bar

### Requirement 7: Run Trawl Button Accessibility

**User Story:** As an investigative analyst, I want the Run Trawl button to remain accessible regardless of which layer I am viewing, so that I can trigger a new scan at any time.

#### Acceptance Criteria

1. THE Alert_Panel SHALL display the "Run Trawl" button in the Headline_Bar actions area, visible across all three layers
2. WHEN a trawl scan completes, THE Headline_Bar SHALL auto-refresh to display updated Viability_Score, Delta, and alert count
3. WHEN a trawl scan completes while Layer 2 is visible, THE Intelligence_Brief SHALL auto-refresh with new indicator data and a new AI brief

### Requirement 8: Briefing API Endpoint

**User Story:** As a frontend developer, I want a dedicated API endpoint for the intelligence brief, so that the brief can be loaded asynchronously without blocking the headline.

#### Acceptance Criteria

1. THE Briefing_API SHALL be accessible at `GET /case-files/{id}/trawl/briefing`
2. THE Briefing_API SHALL return a JSON response containing: `brief_text` (string), `top_entities` (array of strings), `indicator_deltas` (object with before/after per indicator), `generated_at` (ISO timestamp), and `source` ("ai" or "fallback")
3. IF the case_id is not found, THEN THE Briefing_API SHALL return HTTP 404 with error code "NOT_FOUND"
4. THE Briefing_API SHALL be routed through the existing `trawl.py` dispatcher alongside existing trawl routes

### Requirement 9: Deterministic Fallback Completeness

**User Story:** As an investigative analyst, I want all 3 layers to function without any AI service dependency, so that the briefing system works reliably even when Bedrock is unavailable.

#### Acceptance Criteria

1. WHEN Bedrock is unavailable, THE Headline_Bar SHALL display the Viability_Score, Verdict, Delta, alert count, and timestamp using only data from the Impact_API and Alerts_API
2. WHEN Bedrock is unavailable, THE Intelligence_Brief SHALL display the Fallback_Brief text and all 5 Indicator_Bars using data from the Impact_API
3. WHEN Bedrock is unavailable, THE Grouped_Detail_View SHALL function identically to when Bedrock is available, as grouping is performed client-side
4. THE Alert_Panel SHALL indicate the brief source by displaying a subtle "(AI-generated)" or "(auto-summary)" label next to the brief text

### Requirement 10: Existing Functionality Preservation

**User Story:** As an investigative analyst, I want all existing alert panel features (filters, dismiss, investigate, save to notebook, scan history) to continue working within the new 3-layer system.

#### Acceptance Criteria

1. THE Alert_Panel SHALL preserve all existing severity and source filter buttons, applying filters to the alerts displayed in Layer 3 Entity_Cluster groups
2. THE Alert_Panel SHALL preserve the existing dismiss, investigate, and save-to-notebook actions on individual alert cards within Layer 3
3. THE Alert_Panel SHALL preserve the scan history toggle and case evolution timeline features
4. THE Alert_Panel SHALL preserve the `AlertBadgeManager` badge count polling behavior

### Requirement 11: Investigative Entity Titles

**User Story:** As an investigative analyst, I want entity groups in Layer 3 to show descriptive, color-coded titles based on the dominant alert type, so that I can immediately understand the investigative significance of each entity cluster without reading individual alerts.

#### Acceptance Criteria

1. WHEN the Grouped_Detail_View renders an Entity_Cluster header, THE header SHALL display a descriptive title in the format "{Entity Name} — {Descriptor} ({Metric})" instead of just the entity name and count
2. THE descriptor SHALL be determined by the dominant (most frequent) alert_type within the group: "Cross-Case Key Entity" for `cross_case_overlap`, "Network Hub" for `network_expansion`, "Evidence Spike" for `entity_spike`, or "Intelligence Finding" for any other type
3. THE metric text SHALL reflect the dominant type: "{count} overlaps" for cross_case_overlap, "{count} connections" for network_expansion, "{count} new docs" for entity_spike, or "{count} alerts" for default
4. THE descriptor text SHALL be color-coded: blue (#63b3ed) for cross_case_overlap, green (#48bb78) for network_expansion, orange (#f6ad55) for entity_spike, purple (#9f7aea) for default
5. THE investigative title SHALL be computed client-side by the `_buildInvestigativeTitle(group)` method in AlertPanelManager, requiring no additional API calls

### Requirement 12: Per-Entity Intelligence Brief (4-Section Format)

**User Story:** As an investigative analyst, I want each expanded entity group to display a structured BLUF/Key Finding/Critical Gap/Next Action intelligence brief, so that I can quickly assess the investigative significance of each entity without reading every individual alert card.

#### Acceptance Criteria

1. WHEN an Entity_Cluster group is expanded in Layer 3, THE Grouped_Detail_View SHALL render a 4-section intelligence brief above the individual alert cards
2. THE intelligence brief SHALL contain four sections with color-coded left borders: BLUF (green #48bb78), Key Finding (blue #63b3ed), Critical Gap (red #fc8181), and Next Action (amber #ecc94b)
3. THE BLUF section SHALL state the entity name, finding count, linked case names (extracted from alert titles), and evidence document count
4. THE Key Finding section SHALL describe the cross-case significance of the entity
5. THE Critical Gap section SHALL identify missing evidence types (financial, communication records)
6. THE Next Action section SHALL display a "🔍 Search for {entity}" button and a "🕸️ Show Network" button
7. THE intelligence brief SHALL be generated deterministically from alert data without any Bedrock call, using the `_renderEntityIntelBrief(group)` method
8. THE individual alert cards SHALL be hidden by default behind a "📎 View N individual findings" toggle button
9. WHEN the analyst clicks the findings toggle, THE individual alert cards SHALL appear below the intelligence brief, rendered by the existing `_renderCard` method
10. THE findings toggle state SHALL be tracked per entity group via the `_expandedGroupCards` state object

### Requirement 13: Inline Network Graph Visualization

**User Story:** As an investigative analyst, I want to see an inline network graph showing an entity's connections when I click "Show Network" in the entity brief, so that I can visually understand the entity's relationship network without leaving the alert panel.

#### Acceptance Criteria

1. WHEN the analyst clicks the "🕸️ Show Network" button in an entity's Next Action section, THE Alert_Panel SHALL fetch the entity's 1-hop neighborhood from Neptune via `GET /case-files/{graphCaseId}/entity-neighborhood?entity_name={entity}&hops=1`
2. WHILE the network data is loading, THE Alert_Panel SHALL display a loading spinner with "Loading network graph..." text
3. WHEN the network data returns successfully, THE Alert_Panel SHALL render an inline SVG force-directed graph with 300px height inside the entity group
4. THE force-directed graph SHALL use a 100-iteration simulation with repulsion (800 force constant), edge attraction (target distance 60, spring constant 0.05), center gravity (0.01), and velocity damping (0.85)
5. THE graph layout SHALL be deterministic: the same entity name SHALL always produce the same node positions via a seeded random number generator derived from the entity name's character codes
6. THE center entity SHALL be rendered as a larger circle (radius 12) with a white border, fixed at the center of the graph
7. THE neighbor entities SHALL be rendered as smaller circles (radius 8) with subtle borders
8. THE nodes SHALL be color-coded by entity type: person=#48bb78, organization=#63b3ed, location=#f6ad55, date=#9f7aea, with #718096 as the default for other types
9. THE node labels SHALL be truncated to 15 characters with an ellipsis for longer names
10. THE graph SHALL include a color legend below the SVG showing node type colors (Person, Organization, Location, Date, Other)
11. THE graph SHALL include a close button (✕) in the top-right corner to dismiss the visualization
12. WHEN the analyst clicks "Show Network" while a graph is already displayed, THE graph SHALL toggle off (dismiss)
13. WHEN the Neptune API returns zero nodes, THE Alert_Panel SHALL display "No graph data available for this entity" instead of an empty graph
14. WHEN the Neptune API call fails, THE Alert_Panel SHALL display an error message with the failure reason
15. THE graph SHALL render using inline SVG with no external library dependencies
16. THE graph SHALL limit displayed nodes to 20 maximum (first 20 from API response)
