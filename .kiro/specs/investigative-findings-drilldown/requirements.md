# Requirements Document

## Introduction

The Investigative Findings Drilldown redesigns the entity drill-down panel's AI analysis section from shallow graph statistics ("connected to 206 entities") into a 3-level progressive investigation funnel. Level 1 presents AI-generated narrative investigation leads. Level 2 shows the evidence thread supporting a selected lead. Level 3 displays source documents with highlighted passages and the OSINT external research button. The experience mirrors how an investigative journalist follows a thread — each level narrows focus and increases specificity.

## Glossary

- **Lead_Generator**: The backend AI service that analyzes entity context (documents, graph connections, patterns, temporal data) via Amazon Bedrock Claude to produce narrative investigation leads instead of graph statistics
- **Evidence_Assembler**: The backend service that gathers and organizes the supporting documents, entities, timeline entries, and relationship data for a specific investigation lead
- **Drilldown_Panel**: The existing entity drill-down side panel in investigator.html that displays entity intelligence when a user clicks on an entity
- **Investigation_Lead**: A narrative-driven, actionable finding generated by the Lead_Generator, describing WHY something matters investigatively (e.g., "Von Braun appears in 14 documents spanning 1945-1969, with unexplained gaps in 1947-1949")
- **Evidence_Thread**: The collection of documents, entity connections, timeline entries, and key quotes that substantiate a specific Investigation_Lead
- **Source_Viewer**: The Level 3 component that renders actual document excerpts with highlighted relevant passages
- **OSINT_Button**: The "🌐 Research This Externally" button that triggers external web research via the existing OSINT Research Agent
- **Findings_Service**: The existing service (findings_service.py) that persists investigation findings to Aurora PostgreSQL and archives to S3
- **Pattern_Service**: The existing PatternDiscoveryService that discovers entity co-occurrence patterns, centrality patterns, and cross-modal patterns
- **AI_Engine**: The existing InvestigatorAIEngine (investigator_ai_engine.py) that performs case analysis, generates leads, and provides entity neighborhood intelligence

## Requirements

### Requirement 1: Level 1 — Investigation Leads Generation

**User Story:** As an investigative journalist, I want the AI findings section to show narrative investigation leads instead of graph statistics, so that I can immediately identify actionable threads to pursue.

#### Acceptance Criteria

1. WHEN the Drilldown_Panel opens for an entity, THE Lead_Generator SHALL produce between 3 and 7 Investigation_Lead items based on the entity's documents, graph connections, temporal data, and discovered patterns
2. THE Lead_Generator SHALL generate each Investigation_Lead as a narrative sentence that explains WHY the finding matters investigatively, including specific counts, date ranges, entity names, or pattern descriptions
3. THE Lead_Generator SHALL NOT produce leads that only state graph metrics (e.g., "connected to 206 entities") without investigative context explaining significance
4. WHEN the Lead_Generator produces Investigation_Lead items, THE Lead_Generator SHALL assign each lead a lead_type from the set {temporal_gap, cross_case_link, entity_cluster, document_pattern, financial_anomaly, geographic_convergence, relationship_anomaly}
5. WHEN the Lead_Generator produces Investigation_Lead items, THE Lead_Generator SHALL assign each lead a confidence score between 0.0 and 1.0 based on the strength of supporting evidence
6. THE Drilldown_Panel SHALL render Level 1 leads in the section previously occupied by the "⚡ KEY INSIGHTS" section, replacing the graph-statistics-based insights with narrative leads
7. WHEN the user clicks on an Investigation_Lead, THE Drilldown_Panel SHALL transition to Level 2 showing the Evidence_Thread for that lead
8. THE Drilldown_Panel SHALL display a breadcrumb navigation showing the current drilldown level (e.g., "Leads > Evidence Thread > Source Document") to allow the user to navigate back to any previous level

### Requirement 2: Level 1 — Lead Presentation

**User Story:** As an investigative journalist, I want leads presented with visual priority indicators and investigative context, so that I can quickly triage which threads to follow first.

#### Acceptance Criteria

1. THE Drilldown_Panel SHALL render each Investigation_Lead with a color-coded priority indicator: red for confidence ≥ 0.8, amber for confidence ≥ 0.5, green for confidence below 0.5
2. THE Drilldown_Panel SHALL display the lead_type as a category badge on each Investigation_Lead card
3. THE Drilldown_Panel SHALL sort Investigation_Lead items by confidence score in descending order
4. WHEN the entity is part of a known case type (e.g., Ancient Aliens), THE Lead_Generator SHALL tailor lead narratives to the investigative domain (e.g., "Operation Paperclip cluster: 8 German scientists appear together across 12 episodes" or "Geographic convergence: Egypt, Peru, Cambodia mentioned in same context 23 times")

### Requirement 3: Level 2 — Evidence Thread Display

**User Story:** As an investigative journalist, I want to see the actual documents, entities, and connections that support a lead, so that I can evaluate the strength of the evidence before going deeper.

#### Acceptance Criteria

1. WHEN the user selects an Investigation_Lead, THE Evidence_Assembler SHALL retrieve all documents that contain mentions of the entities referenced in that lead
2. WHEN the Evidence_Assembler retrieves documents, THE Evidence_Assembler SHALL extract key quotes (up to 200 characters each) from each document that are relevant to the lead narrative
3. THE Drilldown_Panel SHALL render a timeline of entity mentions showing when each referenced entity appears across the supporting documents, sorted chronologically
4. THE Drilldown_Panel SHALL render a focused relationship map showing only the entities and connections relevant to the selected lead, not the full entity neighborhood
5. WHEN the user clicks on a document or entity in the Evidence_Thread, THE Drilldown_Panel SHALL transition to Level 3 showing the Source_Viewer for that item
6. THE Evidence_Assembler SHALL return the evidence thread data within 5 seconds for leads referencing up to 20 documents
7. THE Drilldown_Panel SHALL display a summary count showing the number of supporting documents, unique entities, and date range covered by the evidence thread

### Requirement 4: Level 3 — Source Documents and External Research

**User Story:** As an investigative journalist, I want to see actual document excerpts with highlighted passages and have the option to research externally, so that I can verify findings and expand the investigation at the point where I have maximum context.

#### Acceptance Criteria

1. WHEN the user selects a document from the Evidence_Thread, THE Source_Viewer SHALL display the document excerpt with the relevant passages highlighted using a distinct background color
2. THE Source_Viewer SHALL display document metadata including filename, ingestion date, and the entities extracted from that document
3. THE Drilldown_Panel SHALL render the OSINT_Button at Level 3 with the label "🌐 Research This Externally", pre-populated with the entity context and lead narrative from the current drilldown path
4. WHEN the user clicks the OSINT_Button at Level 3, THE Drilldown_Panel SHALL trigger the existing OSINT Research Agent (POST /case-files/{id}/osint-research) with research_type "entity" and context including the entity name, lead narrative, and supporting entity names
5. THE Drilldown_Panel SHALL render a "📌 Save to Case" button at Level 3 that saves the current lead, evidence thread, and source document reference as a finding via the existing Findings_Service
6. WHEN the user clicks "📌 Save to Case", THE Findings_Service SHALL persist the finding with finding_type "investigation_lead", the lead narrative as the title, the evidence thread summary as the full_assessment, and the relevant entity_names

### Requirement 5: Backend Lead Generation API

**User Story:** As a frontend developer, I want a dedicated API endpoint for generating investigation leads, so that the drilldown panel can request narrative leads asynchronously.

#### Acceptance Criteria

1. THE Lead_Generator SHALL expose a POST endpoint at /case-files/{id}/entity-leads that accepts entity_name, entity_type, neighbors, and doc_excerpts in the request body
2. WHEN the Lead_Generator receives a valid request, THE Lead_Generator SHALL return a JSON response containing an array of Investigation_Lead objects, each with fields: lead_id, narrative, lead_type, confidence, supporting_entity_names, document_count, and date_range
3. IF the request body is missing the required entity_name field, THEN THE Lead_Generator SHALL return HTTP 400 with a descriptive error message
4. IF the Lead_Generator encounters a Bedrock invocation failure, THEN THE Lead_Generator SHALL return a degraded response containing at least 2 fallback leads generated from graph statistics with narrative framing

### Requirement 6: Backend Evidence Thread API

**User Story:** As a frontend developer, I want a dedicated API endpoint for assembling evidence threads, so that Level 2 can load supporting evidence for a selected lead.

#### Acceptance Criteria

1. THE Evidence_Assembler SHALL expose a POST endpoint at /case-files/{id}/evidence-thread that accepts lead_id, entity_names, lead_type, and narrative in the request body
2. WHEN the Evidence_Assembler receives a valid request, THE Evidence_Assembler SHALL return a JSON response containing: documents (array of {doc_id, filename, excerpt, key_quotes, relevance_score}), entities (array of {name, type, mention_count}), timeline (array of {date, event_description, source_doc}), and relationship_edges (array of {source, target, relationship_type, weight})
3. IF the request body is missing required fields, THEN THE Evidence_Assembler SHALL return HTTP 400 with a descriptive error message
4. THE Evidence_Assembler SHALL limit the response to the top 20 most relevant documents and top 30 entities to maintain response performance

### Requirement 7: Drilldown Navigation and State

**User Story:** As an investigative journalist, I want smooth navigation between drilldown levels with the ability to go back, so that I can explore multiple threads without losing my place.

#### Acceptance Criteria

1. THE Drilldown_Panel SHALL maintain a navigation stack of visited levels so the user can navigate back to any previous level using the breadcrumb
2. WHEN the user clicks a breadcrumb level, THE Drilldown_Panel SHALL restore the view for that level without re-fetching data from the backend
3. THE Drilldown_Panel SHALL animate transitions between levels using a horizontal slide animation (left for deeper, right for back) completing within 300 milliseconds
4. WHILE the Evidence_Assembler or Lead_Generator is loading data, THE Drilldown_Panel SHALL display a contextual loading indicator with a description of what is being loaded (e.g., "Assembling evidence thread for 'Operation Paperclip cluster'...")

### Requirement 8: OSINT Button Placement

**User Story:** As an investigative journalist, I want the OSINT external research button to appear at the bottom of the investigation funnel where I have maximum context, so that my external searches are targeted and meaningful.

#### Acceptance Criteria

1. THE Drilldown_Panel SHALL render the OSINT_Button at Level 3 (Source Documents) as a prominent action button
2. THE Drilldown_Panel SHALL NOT render the OSINT_Button at Level 1 (Investigation Leads) or Level 2 (Evidence Thread)
3. WHEN the OSINT_Button is rendered at Level 3, THE Drilldown_Panel SHALL pre-populate the research context with the full drilldown path: entity name, lead narrative, evidence thread summary, and current document reference
4. THE Drilldown_Panel SHALL continue to render the existing OSINT_Button in the "🌐 EXTERNAL INTELLIGENCE" section of the entity drill-down panel independently of the 3-level drilldown flow
